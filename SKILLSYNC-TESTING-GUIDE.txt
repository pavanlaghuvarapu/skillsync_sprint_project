========================================================
ARCHITECTURE OVERVIEW
========================================================

Infrastructure Services:
  config-server        -> 8888 (GitHub-backed centralized config)
  eureka-server        -> 8761 (Service discovery)
  api-gateway          -> 8080 (JWT validation + routing)

Business Services:
  auth-service         -> 8081 (JWT login/register/logout + Redis)
  user-service         -> 8082 (profiles + image upload)
  mentor-service       -> 8083 (mentor management + Redis cache)
  session-service      -> 8084 (booking + RabbitMQ + Feign + Circuit Breaker)
  notification-service -> 8085 (RabbitMQ consumer + Feign)
  review-service       -> 8086 (reviews + Feign + Circuit Breaker)
  group-service        -> 8087 (peer learning groups)

Supporting:
  PostgreSQL  -> 5432 (separate DB per service)
  RabbitMQ    -> 5672 (async notifications)
  Redis       -> 6379 (JWT + mentor cache)

========================================================
CONCEPTS DEMONSTRATED IN THIS PROJECT
========================================================

1. Config Server     -> GitHub-backed, all services load config from it
2. Eureka Server     -> All services register, Feign uses service names
3. API Gateway       -> JWT validation filter, routes all requests
4. JWT Auth          -> Generated in auth-service, validated in every service
5. Role-Based Access -> ROLE_ADMIN, ROLE_MENTOR, ROLE_LEARNER enforced
6. Feign Client      -> session->mentor, session->user, review->mentor, notification->user
7. Circuit Breaker   -> Resilience4j on session->mentor, review->mentor
8. Redis Cache       -> Mentor list, mentor search, skills (auth token store)
9. RabbitMQ          -> Session events published, notification-service consumes
10. Docker           -> Dockerfile per service, docker-compose.yml

STEP 5 - JWT FLOW EXPLANATION
========================================================

Gateway receives request
    |
    v
JwtAuthFilter checks Authorization header
    |
    |-- No token? -> 401 UNAUTHORIZED
    |
    |-- Invalid token? -> 401 UNAUTHORIZED
    |
    v
Token valid -> extract email + role
    |
    v
Forward to downstream service with headers:
  X-Auth-User-Email: pavan@gmail.com
  X-Auth-User-Role: ROLE_LEARNER
    |
    v
Downstream JwtAuthFilter reads headers
    |
    v
Sets SecurityContext with email + role
    |
    v
@PreAuthorize checks role -> allow or deny
